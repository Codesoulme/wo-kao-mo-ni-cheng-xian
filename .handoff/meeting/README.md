## 二、什么时候开会

不是每件事都要开会。开会适用于：

1. **理解对齐**：小虾米给的规则 / 准则 / 项目结构，小薪读完有疑问。
2. **边界确认**：小薪要动的东西触到了"小薪不能自己动"的红线，需要小虾米正式授权。
3. **设计决策**：影响玩家体验、剧情方向、UI 槽位的设计选择。
4. **跨任务对齐**：多个任务之间的依赖、先后、共享资源。
5. **复盘 / 教训**：改坏了或踩坑后，回头确认"以后怎么避免"。

不需要开会：

- 普通小修小补（除非小薪有疑问）。
- 标准实现（按工作准则 + 任务 plan 走）。
- owner 已批准的 commit 后续工作。

## 三、会议流程

```
┌────────────────┐    ┌────────────────┐    ┌────────────────┐    ┌────────────────┐
│  任何一方起会议  │ -> │ 双方写 to-XXX  │ -> │ 双方同步读、回复 │ -> │ 写入 decisions  │
└────────────────┘    └────────────────┘    └────────────────┘    └────────────────┘
                                                                       │
                                                                       ▼
                                                              action-items 落回任务
                                                                       │
                                                                       ▼
                                                              归档到 archive/
```

### 第 1 步：起会议

任何一方（小虾米 / 小薪）在 `<meeting-topic>/agenda.md` 里写：

```markdown
# 会议议程

## 主题
<一句话>

## 发起人
小虾米 / 小薪

## 参与方
- 小虾米
- 小薪
- （一万万 owner 仅在需要拍板设计核心时参与）

## 时间
YYYY-MM-DD HH:MM

## 背景
<为什么开会>

## 要决定 / 澄清 / 同步的事
1. ...
2. ...

## 期望产出
- decisions.md（决议）
- action-items.md（待办）
- （可选）回写到 .handoff/<task>/

## 时限
- 普通：24 小时内出 decisions
- 紧急：立刻开（IM 同步）
- 阻塞：标 [BLOCKED]，等 owner
```

然后更新 `current.md` 指向该会议。

### 第 2 步：双方写 to-XXX

**小虾米**写 `xiaoxia-to-xiaoxin.md`：

```markdown
# 小虾米要交代 / 跟小薪确认的事

## 1. 我要交代给你的项目背景
- ...

## 2. 我想跟你确认我理解对不对
- [ ] 你对 X 的理解是 Y，对吗？
- [ ] 你认为应该 Z，对吗？

## 3. 我让你做 X 之前，需要你明白的边界
- ...

## 4. 我没把握 / 希望你给意见的事
- ...

## 5. 我之前没说清 / 你可能误解的
- ...
```

**小薪**写 `xiaoxin-to-xiaoxia.md`：

```markdown
# 小薪要跟小虾米确认 / 问的事

## 1. 我读完任务 / 准则后的疑问
- Q: ...
- Q: ...

## 2. 我对任务 / 准则的理解
- 我理解 X 是 Y，对吗？

## 3. 我打算这样做，你能接受吗
- 方案 A: ...
- 方案 B: ...

## 4. 我需要你给的东西 / 权限
- 能不能让我看 X 文件？
- 能不能让我改 Y 函数？

## 5. 我发现的疑似问题
- ...
```

### 第 3 步：双方同步读、回复

不是真的要"开会"，而是：

- 小虾米读 `xiaoxin-to-xiaoxia.md`，在 `xiaoxia-to-xiaoxin.md` 末尾"小虾米回复"小节里逐条回。
- 小薪读 `xiaoxia-to-xiaoxin.md`，在 `xiaoxin-to-xiaoxia.md` 末尾"小薪回复"小节里逐条回。
- 反复往返，直到两边都没新问题。

### 第 4 步：写决议

`decisions.md`：

```markdown
# 会议决议

## 已达成共识的事实
- ...

## 已确认的边界
- 小薪可以做：...
- 小薪不能做：...

## 已确认的做法
- ...

## 仍未达成共识（升级给 owner）
- ...
```

### 第 5 步：写 action-items

`action-items.md`：

```markdown
# 会议产生的待办

## 给小薪的（落回 .handoff/<task>/）
- [ ] ...

## 给小虾米的
- [ ] ...

## 给 owner 的（如需）
- [ ] ...
```

### 第 6 步：归档

会议结束后，`current.md` 清空，会议目录移到 `archive/`：

```
archive/YYYY-MM-DD-<topic>.md
```

整个会议目录打包成一份归档 markdown 保留。

## 四、状态机

| 状态 | 标记 | 说明 |
|---|---|---|
| 已发起 | `[AGENDA]` | agenda.md 已写，等双方写 to-XXX |
| 同步中 | `[SYNC]` | 双方在反复读 + 回复 |
| 待决议 | `[DECIDING]` | 同步完成，等写入 decisions |
| 已决议 | `[DECIDED]` | decisions.md + action-items.md 都已写 |
| 已归档 | `[ARCHIVED]` | 已移到 archive/ |
| 升级给 owner | `[ESCALATED]` | 仍存疑的部分需 owner 拍板 |
| 阻塞 | `[BLOCKED]` | 任一方超过 24 小时未回复 |

`current.md` 顶部必须有 `[状态]` 标签。

## 五、纪律

### 5.1 会议期间

- 会议期间**不直接动代码**。
- 会议期间**不开新任务**。
- 会议期间**不刷 commit / push**。

### 5.2 谁有权起会议

- 小虾米：随时。
- 小薪：当以下情况发生时**必须**起会议而不是直接动：
  - 任务 plan 里说"由小虾米决定"的事项。
  - 发现疑似违背工作准则的现状。
  - 不确定某个边界是不是真红线。
  - 想改 `engine.ts` / `llm.ts` prompt 主体 / `types.ts` schema / 真实存档 / Git 历史。
  - 工作准则与任务 plan 冲突。

### 5.3 时限

- 普通会议：24 小时内出 decisions。
- 紧急会议：立刻开（IM 同步），但**仍要落档**到 `<meeting-topic>/agenda.md`。
- 阻塞会议：标 `[BLOCKED: reason]`，等 owner 决定。

### 5.4 与 .handoff/<task>/ 的关系

- 会议**不替代**任务 plan / review / verification 流程。
- 会议**产生**决议，落回任务 plan：
  - 把决议写到对应任务 `xiaoxin-plan.md` 头部"会议决议参考"。
  - 把 action-items 落回 `task.md` 或新建子任务。

### 5.5 与工作准则的关系

- 会议**不能推翻**工作准则的硬性边界。
- 会议**可以澄清**工作准则没说清的细节。
- 会议**不能授权**：
  - 改真实玩家存档（必须 owner 拍板）
  - 改 Git 历史（必须 owner 拍板）
  - 引入新外部依赖（必须 owner 拍板）
  - 改 `types.ts` schema（必须小虾米 + owner 共同拍板）
