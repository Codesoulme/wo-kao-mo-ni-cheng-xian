## 1. 项目定位

本文档基于用户此前提供的《修仙模拟器统一技术设计文档 v1.0》（2026-06-17）与当前 Next.js 代码实现整理。

《我靠模拟成仙》是一款 **AI 驱动的修仙人生模拟 RPG**。玩家不只是点击固定事件，而是在一个会记忆、会延续因果、会根据角色状态改写后续命运的修仙世界里，陪伴角色从凡人一步步走向求道、争命、破境、飞升或陨落。

核心目标不是做传统数值挂机游戏，而是做一个“可被 AI 持续导演、但由规则引擎兜底裁判”的修仙模拟器：

- AI 负责生成叙事、因果、人物牵挂、秘境线索、战斗缘由、拍卖会过程等内容。
- 引擎负责校验属性变化、寿元、境界、物品、战斗、突破、状态、储物容量等规则。
- UI 负责把复杂系统包装成沉浸式修仙体验，避免暴露现实实现词和内部字段。

## 2. 设计原则

### 2.1 AI 优先，但规则裁判

游戏新增功能必须优先考虑是否能被 AI 生成、调整或驱动。AI 可以提出事件和变化，但最终由本地引擎执行、限制和校验。

原则：

- AI 可以生成剧情、选择、状态、物品、敌人、秘境、战利品、后续牵挂。
- AI 不直接拥有最终裁判权，不能随意修改年龄、越权改规则、强制灌入不合理数值。
- 所有 AI 输出必须可解析、可校验、可回退。

### 2.2 沉浸优先

玩家界面不能出现出戏的系统词，例如：AI、id、调试、缓存、预演、预加载、失效、接口、JSON。这些机制可以在幕后存在，但面向玩家时必须转译为修仙世界内的语言，例如“天机未明”“因果未定”“灵机牵引”“旧缘回响”等。

### 2.3 因果连续

游戏必须避免“每年一个孤立短故事”。重要事件应形成长期因果：

- 未了因果会进入 pendingThreads。
- 角色牵挂会在后续年份自然回响。
- 同一年发生的“稍后/夜里/几日后”事件应拆成多个史册片段，而不是丢失后续。
- 战斗、秘境、拍卖、突破都应承接触发事件的标题和叙事。

### 2.4 不显示无意义信息

玩家界面不展示 0 值变化、未获得的东西、没有实际变化的奖励、空状态、内部英文 key、纯调试信息。

例如 `heartDemon +2` 应显示为 `心魔 +2`，`cultivationExp 0` 不应显示。

## 3. 核心体验循环

### 3.1 开局

玩家进入游戏后创建角色：

1. 可填写角色名，也可留空由游戏生成。
2. 系统随机灵根、五行倾向、出身背景。
3. AI 根据随机结果生成出生叙事。
4. 角色进入 0 岁史册。

开局必须依赖可用 AI 配置；缺少配置时应提示玩家配置，而不是长期靠本地 fallback 硬玩。

### 3.2 年份推进

每次推进代表角色人生前进一年，基本流程：读取角色状态 → 挑选事件蓝图 → 汇总上下文 → 调用 AI 生成事件 → 引擎校验并应用结果 → 写入史册。

年份推进不能出现“无事发生”的空年。即使普通年份，也应有小收获、小挫折、小牵挂或状态演变。

### 3.3 玩家干扰

玩家可以输入自然语言干预世界。系统按输入类型处理：行动、对话、越权、规则操纵。越权和规则操纵必须被世界观内拒绝，不能让玩家直接篡改规则。

### 3.4 选择节点

重大事件可以触发选择。选择必须有真实后果：改变属性或状态、推进/完成/失败因果、触发战斗、获得或失去物品、开启后续选择链。

选择不应只是“换一段文本”。

### 3.5 战斗

战斗由事件触发，不应凭空出现。战斗体验规则：

1. 先展示战斗缘由，让玩家点击“入战”。
2. 战斗标题和缘由承接触发事件。
3. 回合记录应像小说片段，而不是机械伤害清单。
4. 胜利后根据敌人随身身家、物品、储物袋、灵石等生成战利品。
5. 战斗结束先停留结果态，不要闪回叙事页。

### 3.6 突破

允许一朝顿悟连破多重境界，但必须有充分由头，例如大机缘、高阶丹药、秘境传承、雷劫洗礼、长期铺垫后的顿悟。

不能在资质平平、无奇遇、无足够修为时乱跳境界。

突破展示规则：

- “破境/突破”标签只在最终突破成功叙事上显示。
- 准备、冲关、失败、临门一脚不能提前显示突破成功标签。

## 4. 系统模块

### 4.1 AI 配置系统

当前游戏内置 AI 配置入口，负责保存 Base URL、API Key、模型名、可选 Chat ID / User ID。

设计要求：配置保存在本地，不提交仓库；前端只脱敏显示 key；提供“测试连接”按钮；开局、推进、选择、对话等主流程在未配置时应提示配置。

### 4.2 角色状态系统

角色基础状态包含姓名、性别、年龄、寿元、灵根、五行、境界、修为、气血、灵力、攻防速、幸运、悟性、灵石、声望、门派、师承、位置、心魔、当前状态、物品、装备、储物容量、长期记忆、未了因果、角色意图、战斗会话、灵宠、秘境探索记录。

状态不是装饰，而是 AI 生成事件时必须参考的上下文。

### 4.3 境界系统

默认境界体系：凡人 → 炼气 → 筑基 → 金丹 → 元婴 → 化神 → 大乘 → 渡劫 → 飞升。

但默认体系不是唯一法则。AI 可以在有充分因果时通过境界画像覆盖显示，例如“炼气九十九层”“三转金丹”“伪元婴”“剑胎初成”。

要求：底层保持稳定 realm 字段用于规则计算；显示层可以由 realmProfile 修饰；特殊境界必须有来源和解释。

### 4.4 状态与牵挂系统

状态分为属性状态、技能状态、增益、减益、特殊命格/体质/传承、身份、任务/因果、环境影响。

展示规则：顶部状态按最近获得顺序从左到右显示；状态、线索、角色意图应分类、折叠、按时效呈现；重要牵挂必须能在后续年份回响。

### 4.5 物品与装备系统

物品类型：武器、防具、饰物、法宝、丹药、材料、工具、功法。

装备不是固定 5 个槽位。玩家可以佩戴多个戒指、法宝或特殊装备，合理性由 AI 和叙事判断。

储物容量规则：初始容量有限；储物袋等工具获得后自动扩容；储物袋不应作为普通装备佩戴来生效。

物品界面不能暴露内部 item id。

### 4.6 修炼速度系统

修炼速度由引擎统一归算，来源包括灵根倍率、装备/功法加成、状态加成、心魔惩罚、灵宠被动。

规则：UI 应展示多个来源；来源过多时可折叠分组；卖掉、卸下、消耗物品后，加成必须同步清理。

### 4.7 心魔系统

心魔是反向属性：心魔增加是负面，应显示红色；心魔减少是正面，应显示绿色。心魔高时影响修炼速度，并可能触发心魔战、失控或走火风险。

### 4.8 战斗系统

战斗由 CombatSession 管理，包含敌人列表、当前敌人、回合数、战斗日志、玩家临时战斗属性、可用法术/功法、可用丹药/符箓、灵宠参战状态、胜负状态。

战斗战利品规则：敌方修士应有随身身家；胜利后可获得储物袋、装备、丹药、功法、灵石等；妖兽应掉落材料、内丹、爪牙等；战利品不能只是叙事里提到，必须进入物品系统。

### 4.9 坊市与拍卖系统

坊市/拍卖不应是常驻无因果入口。入口应由事件或因果触发。

拍卖设计：轻量入口只提示有拍卖因缘；真正进入后必须有完整拍卖场景；拍品要有来历、价格、竞价者、风险和后续影响；可通过连续选择推进报价、观望、放弃、争夺等。

### 4.10 秘境系统

秘境入口只显示因果中已经显露的地点，不显示无关静态收费列表。

秘境命名规则：应提炼为真实地点名，例如“青岚江雾楼”“倒悬钟楼”；避免显示“或可探明雾楼”这类句子残片。

秘境前置条件：信物检测应宽松匹配。例如需求“玉片/潮湿玉片/信物”时，持有“潮禁玉片”等同源物也应可入。

### 4.11 灵宠系统

灵宠可以由 AI 在合适事件中生成，例如救助幼兽、斩妖后收服、秘境传承、坊市购买、前人遗赠。

灵宠属性包括种族、名称、稀有度、境界、气血、攻击、防御、速度、五行、忠诚、饱食、等级、经验、技能。

灵宠可在战斗中协助，但不应每年机械刷存在感。

### 4.12 阵法、炼丹、符箓

这些系统应与 AI 联动：炼丹应有材料、成功率、成品、失败产物；阵法应来源于阵盘或传承，产生可持续状态；符箓应作为战斗内可使用消耗品，具备攻击、防御、治疗、遁逃、镇压等效果。

## 5. UI/交互规范

### 5.1 文案规范

所有玩家可见文案必须中文化，不能显示内部英文 key。

示例：

- `heartDemon` → 心魔
- `cultivationExp` → 修为
- `storageCapacity` → 储物容量
- `activeStatuses` → 当前状态 / 缘法 / 命格等视语境而定

### 5.2 长文本适配

修仙叙事天然长文本，UI 必须支持弹窗滚动、标题和关闭按钮不重叠、移动端自适应、长状态/长物品描述折叠或换行、史册默认展开最近事件。

### 5.3 信息分层

建议分层：顶部显示年龄、境界、气血/灵力、近期重要状态；角色页显示基础属性、境界、修炼速度、五行、心魔；宝物页显示储物、装备、习得法术；缘法页显示长期状态、命格、牵挂、线索；史册显示事件、战斗、选择、突破、秘境等人生记录。

## 6. 数据与接口设计概览

### 6.1 主要 API

当前项目包含以下主要接口：

- `/api/ai-config`：读取/保存 AI 配置
- `/api/ai-config/test`：测试 AI 连接
- `/api/game/new`：新建角色
- `/api/game/state`：读取当前状态
- `/api/game/advance`：推进一年
- `/api/game/preload-advance`：幕后生成下一年候选
- `/api/game/choose`：处理选择结果
- `/api/game/interfere`：处理玩家干扰
- `/api/game/combat/action`：战斗行动
- `/api/game/combat/end`：战斗结算
- `/api/game/item`：物品使用、装备、卸下
- `/api/game/market`：坊市/交易
- `/api/game/exploration`：秘境探索
- `/api/game/alchemy`：炼丹
- `/api/game/formation`：阵法
- `/api/game/pet`：灵宠

### 6.2 AI 输出校验

涉及 AI JSON 的功能必须检查：JSON 可解析；字段符合 schema；属性名在允许列表内；数值在合理范围内；物品类型合法；状态分类合法；战斗敌人和战利品结构合法；不出现内部字段泄漏到玩家 UI。

修改 prompt、schema、上下文、调用链后，必须真实调用本地 AI 配置验证 JSON 可解析且能通过引擎校验。

## 7. 当前版本已具备能力

当前版本已经具备：游戏内 AI 配置与连接测试、角色出生生成、年份推进、玩家自然语言干扰、选择节点、事件蓝图、未了因果、角色意图、境界与突破、境界画像覆盖、心魔、战斗与战斗结算、战利品生成、储物容量、装备/功法/物品、修炼速度来源展示、坊市与拍卖选择链、秘境入口与探索、炼丹、阵法、灵宠、史册与状态展示。

## 8. 后续优化方向

### 8.1 短期

- 梳理所有玩家可见文案，继续清理系统感词汇。
- 加强秘境、拍卖、战斗的因果承接。
- 为核心 AI 调用补充更稳定的回归测试脚本。
- 完善设计文档中的数值表、状态表、事件蓝图表。

### 8.2 中期

- 增加门派关系网。
- 增加 NPC 长期记忆。
- 增加更完整的修仙世界地图。
- 加强物品合成、炼器、功法参悟。
- 做更系统的存档/读档管理。

### 8.3 长期

- 多角色传承。
- 家族/宗门兴衰模拟。
- 更强的世界线因果网。
- 结局谱系：飞升、坐化、入魔、立宗、转世、遗蜕传承等。

## 9. 设计验收清单

新增功能上线前检查：

- [ ] 是否能被 AI 生成、调整或驱动？
- [ ] 是否有本地引擎校验？
- [ ] 是否会破坏因果连续？
- [ ] 是否有玩家可见英文 key 或内部 id？
- [ ] 是否出现 AI/调试/缓存/预演等出戏词？
- [ ] 是否展示了 0 值或无意义信息？
- [ ] 是否在移动端长文本下可读？
- [ ] 是否真实调用 AI 验证 JSON 可解析？
- [ ] 是否通过 TypeScript、Lint、Build？

## 10. 文档维护规则

- 代码变更涉及玩法、系统、AI schema、UI 规则时，应同步更新本文档。
- 文档应描述“玩家体验 + 系统规则 + AI 边界”，不要只写代码实现。
- 基准版不删除，作为升级对照。

## 11. 架构进展记录（2026-06-19）

本轮升级把项目从“可玩的 AI 原型”推进到“可注册、可追踪、可审计的长期世界模拟原型”。当前已经落地的 Lite 架构层包括：

- ContentRegistry Lite：AI 生成的物品、状态、线索、NPC、事件等内容先注册、再进入世界。
- NPC Persistence Lite：结构化 NPC 能进入角色状态，拍卖竞拍者、战斗敌人和剧情人物可以持续存在。
- CausalGraph Lite：事件、物品、状态、NPC、秘境、战斗、选择、灵宠之间的来源关系进入因果图。
- EffectResolver / ERPE Lite：属性变化统一解析，记录应用、拒绝、边界裁剪和警告。
- QuestEntry Lite：由 pendingThreads 派生任务/线索索引，供 AI 和调度器参考。
- AI Boundary Validator Lite：对越界引用、过大数值、未承接高优先级线索等情况生成 warning/audit。
- StateChangeLog Lite：主要行动都会生成隐藏状态变更审计。
- WorldFacts Lite：地点、NPC、势力、秘境等长期事实保守持久化。
- Event Scheduler Lite：根据未决线索、期限、意图和状态生成下一次 AI 事件的调度提示。
